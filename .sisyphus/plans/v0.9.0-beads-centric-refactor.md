# v0.9.0: Beads-Centric 架构重构

## 目标

完全抛弃 `task-map.yaml`，以 Beads 作为任务的唯一真理源。Agent 不操作文本文件，
所有操作通过统一 CLI 入口 `flowforge` 命令走 `TaskBackend` 接口。

## 核心变化

| 方面 | v0.8 | v0.9 |
|------|------|------|
| 任务真理源 | task-map.yaml（YAML 文件） | Beads 后端 |
| 任务操作方式 | 12 个独立 task-*.js 脚本 | `flowforge task <action>` |
| Context 加载 | 4 个独立 context 脚本 | `flowforge <skill>-context` |
| Beads 同步 | 双写 + hooks 双向对账 | 无需同步（beads 唯一真理源） |
| 脚本入口 | `node .flowforge/scripts/*.js` | `flowforge <command>` |
| 人类可读 | task-map.yaml（Agent 也操作它） | tasks.snapshot.md（纯展示） |

## Phase 0: 统一 CLI 设计 (`flowforge` 命令)

### 入口

```bash
# 安装在目标项目的 .flowforge/ 下
.flowforge/bin/flowforge <command> [options]

# 用法模式
flowforge <command> [--project <root>] [--proposal <id>] [args...]
```

`--project` 默认 `.`（当前目录）。`--proposal` 自动搜索活跃 proposal。

| 方面 | 之前 (node scripts/*.js) | 之后 (flowforge) |

// ============================================================

class BackendCapabilities {
  /** 认领是否原子（多 Agent 安全） */

  status = 'pending';  // pending | in_progress | done | blocked | cancelled

  summary = '';

### 方法契约

```typescript
interface TaskBackend {
  // ── 能力 ──
  getCapabilities(): BackendCapabilities;

  // ── 生命周期 ──
  init(proposalId: string, title: string): Promise<{ epicId: string }>;
  teardown(proposalId: string): Promise<void>;

  // ── 任务创建 ──
  addTask(proposalId: string, task: TaskDef): Promise<{ taskId: string }>;
  addTasks(proposalId: string, tasks: TaskDef[]): Promise<{ taskIds: string[] }>;
  discoverTask(proposalId: string, parentTaskId: string, task: TaskDef): Promise<{ taskId: string }>;
  cancelTask(proposalId: string, taskId: string, reason?: string): Promise<void>;

  // ── 状态流转 ──
  claimTask(proposalId: string, taskId: string): Promise<{ claimed: boolean; conflict?: string }>;
  completeTask(proposalId: string, taskId: string, summary?: string): Promise<void>;
  blockTask(proposalId: string, taskId: string, reason: string): Promise<void>;
  unclaimTask(proposalId: string, taskId: string): Promise<void>;
  reopenTask(proposalId: string, taskId: string): Promise<void>;

  // ── 依赖管理 ──
  addDependency(proposalId: string, taskId: string, dependsOnTaskId: string): Promise<void>;
  removeDependency(proposalId: string, taskId: string, dependsOnTaskId: string): Promise<void>;

  // ── 查询 ──
  getReadyTasks(proposalId: string): Promise<Task[]>;
  getStatus(proposalId: string): Promise<TaskStatus>;
  getBlockedTasks(proposalId: string): Promise<Task[]>;
  getTask(proposalId: string, taskId: string): Promise<Task | null>;
  isAllDone(proposalId: string): Promise<boolean>;

  // ── 快照导出 ──
  exportSnapshot(proposalId: string): Promise<string>;
}
```

### 与旧接口的关键差异

| 旧 (TaskAdapter) | 新 (TaskBackend) | 说明 |
|---|---|---|
| 每个方法首参 `proposalDir` | 每个方法首参 `proposalId` | 后端自行定位，不暴露文件路径 |
| 返回含 `conflict` error msg | 同上 | 保留 |
| `_readTaskMap` / `_writeTaskMap` | 不存在 | 不再 touch 文件 |
| `getContext()` | 删除 | 归 context 脚本负责 |
| `sync()` | 删除 | 不需要同步 |
| `cleanup()` | → `teardown()` + `isAllDone()` | 职责分离 |
| 无 `addTasks`（批量） | 新增 | Design SKILL 需要 |
| 无 `reopenTask` | 新增 | 回退场景需要 |
| 无 `addDependency` / `removeDependency` | 新增 | 需求树演化需要 |
| 无 `isAllDone` | 新增 | 结束条件判断 |

## Phase 2: BeadsBackend (`lib/backends/beads.js`)

薄封装 `bd` CLI，不直接耦合 proposal 目录结构。

### 关键实现

```javascript
class BeadsBackend {
  constructor(projectRoot) {
    this.root = projectRoot;
    this.epics = new Map();  // proposalId → epicId 缓存
  }

  // bd 命令包装
  _bd(args) {
    return execSync(`bd ${args}`, {
      cwd: this.root,
      encoding: 'utf8',
      stdio: 'pipe',
      timeout: 30000
    }).trim();
  }

  async init(proposalId, title) {
    const result = this._bd(
      `create "${escape(title)}" --type epic --labels proposal:${proposalId} --json`
    );
    const epic = JSON.parse(result);
    this.epics.set(proposalId, epic.id);
    return { epicId: epic.id };
  }

  async addTasks(proposalId, tasks) {
    const epicId = await this._resolveEpic(proposalId);
    const ids = [];
    for (const task of tasks) {
      const labels = this._buildLabels(task, proposalId);
      const result = this._bd(
        `create "${escape(task.title)}" --type task --parent ${epicId} ` +
        `--labels ${labels} --description "${escape(task.description || '')}" --json`
      );
      ids.push(JSON.parse(result).id);
    }
    // 批量创建依赖链接
    for (let i = 0; i < tasks.length; i++) {
      if (tasks[i].dependencies) {
        for (const depIdx of tasks[i].dependencies) {
          await this._bd(`link ${ids[i]} ${ids[depIdx]}`);
        }
      }
    }
    return { taskIds: ids };
  }

  async getReadyTasks(proposalId) {
    // 利用 beads 原生拓扑排序
    const epicId = await this._resolveEpic(proposalId);
    const result = this._bd(`ready --json`);
    return JSON.parse(result)
      .filter(t => t.labels?.includes(`proposal:${proposalId}`))
      .map(this._toTask);
  }

  async claimTask(proposalId, taskId) {
    try {
      this._bd(`update ${taskId} --claim`);
      return { claimed: true };
    } catch (e) {
      return { claimed: false, conflict: extractError(e) };
    }
  }

  async completeTask(proposalId, taskId, summary) {
    const reason = summary ? ` --reason "${escape(summary)}"` : '';
    this._bd(`close ${taskId}${reason}`);
  }

  async getStatus(proposalId) {
    const epicId = await this._resolveEpic(proposalId);
    const result = this._bd(`list --label proposal:${proposalId} --all --json`);
    const tasks = JSON.parse(result).map(this._toTask);
    // 聚合统计...
  }

  async exportSnapshot(proposalId) {
    const status = await this.getStatus(proposalId);
    return formatMarkdownTable(status.tasks);
  }

  getCapabilities() {
    return {
      atomicClaim: true,
      discoveredFrom: true,
      auditTrail: true,
      dependencySort: true
    };
  }

  _buildLabels(task, proposalId) {
    return [
      `type:${task.type || 'implementation'}`,
      `proposal:${proposalId}`,
      ...(task.epic || []).map(e => `epic:${e}`),
      ...(task.sourceTasks || []).map(s => `source:${s}`),
    ].join(',');
  }

  _toTask(beadIssue) {
    return {
      id: beadIssue.id,
      title: beadIssue.title || '',
      description: beadIssue.description || '',
      type: beadIssue.labels?.find(l => l.startsWith('type:'))?.replace('type:', '') || 'implementation',
      status: mapBeadStatus(beadIssue.status),
      dependencies: beadIssue.depends_on || [],
      sourceTasks: beadIssue.labels?.filter(l => l.startsWith('source:')).map(l => l.replace('source:', '')) || [],
      epic: beadIssue.labels?.filter(l => l.startsWith('epic:')).map(l => l.replace('epic:', '')) || [],
      createdAt: beadIssue.created_at || '',
      updatedAt: beadIssue.updated_at || '',
    };
  }
}
```

## 升级路径 (0.8 → 0.9)

### 关键：解决当前项目中的 beads 同步 Bug

当前 v0.8 项目存在已知问题：`task-map.yaml` 和 beads 不同步——有些只更新了 YAML 没写 beads，有些反之。升级脚本必须处理这个。

### 升级步骤

```bash
# 1. 运行升级安装（更新托管文件 + 创建 ./flowforge CLI + 安装新 hooks）
./scripts/install.sh upgrade /path/to/target/project

# 2. 升级脚本自动执行：
#    a. 更新 SKILL、脚本、schema、指南
#    b. 创建根目录 ./flowforge 薄封装
#    c. 更新 AGENTS.md
#    d. 更新 config.yaml (adapter: yaml → beads)
#    e. 安装新 hooks（snapshot 刷新，替代旧同步 hooks）
#    f. 删除旧 task-*.js 脚本和 lib/adapters/

# 3. 手动执行（对每个活跃 proposal）：
cd /path/to/target/project

# 3a. 清理 beads 中的孤儿 issue
for CR_ID in $(ls ff-wiki/workspace/proposals/active/ 2>/dev/null); do
  ./flowforge upgrade cleanup-orphans --proposal $CR_ID
done

# 3b. 将 task-map.yaml 中的已有任务同步到 beads
for CR_ID in $(ls ff-wiki/workspace/proposals/active/ 2>/dev/null); do
  ./flowforge upgrade migrate-from-yaml --proposal $CR_ID
done

# 3c. 验证同步结果
for CR_ID in $(ls ff-wiki/workspace/proposals/active/ 2>/dev/null); do
  ./flowforge task status --proposal $CR_ID
done

# 3d. 删除 task-map.yaml，生成只读快照
for CR_ID in $(ls ff-wiki/workspace/proposals/active/ 2>/dev/null); do
  ./flowforge task snapshot --proposal $CR_ID
  rm ff-wiki/workspace/proposals/active/$CR_ID/task-map.yaml
done

# 4. 验证
./flowforge task all-done --proposal <CR-id>
```

### 迁移脚本 `migrate-from-yaml` 行为

```
1. 读取 task-map.yaml
2. 解析所有任务（id, title, description, type, status, dependencies, sourceTasks, epic）
3. 创建 beads epic（如不存在）
4. 为每个任务创建 beads issue，设置标签和依赖
5. 根据 YAML 中的 status 设置 beads issue 状态（pending→open, in_progress→claim, done→close）
6. 输出迁移统计：共 N 个任务，成功创建 M 个，跳过（已存在）K 个
```

### `cleanup-orphans` 行为

```
1. 查询 beads 中所有带 proposal:<CR-id> 标签的 issue
2. 读取 task-map.yaml（如果存在）获取期望的任务列表
3. 关闭 beads 中不存在于 task-map.yaml 的 orphan issue
4. 输出清理统计
```
