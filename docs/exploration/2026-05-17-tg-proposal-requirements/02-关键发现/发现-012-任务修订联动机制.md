# 发现-012：任务修订联动机制

**状态**: 已确认
**发现时间**: 2026-05-17
**来源**: Karl Wiegers, Agile Scrum Best Practices, GitHub 代码实现

---

## 核心发现

### Reopen 黄金法则

> **只有未满足 DoD 的任务才应该 reopen。已满足 DoD 的任务需要修改时，应创建新任务。**

---

## 任务状态转换规则

```
pending → in_progress → complete
    ↑                      ↓
    └────── reopen ────────┘
    
转换条件：
- complete → pending: 只有未满足 DoD 时允许
```

---

## 提案修订时的任务联动决策

### 已完成任务

```
if (满足 DoD && 需求变更) {
  创建 revision-task:
    - parent_task_id = 原任务ID
    - 标签: revision-task
    - 说明: "基于提案 v1.1 的修订需求"
}
else if (未满足 DoD) {
  reopen 原任务:
    - 状态: pending
    - reopen_reason: "提案修订 v1.0→v1.1"
}
```

### 进行中任务

```
if (小调整) {
  更新任务:
    - 描述/验收标准
    - 记录变更历史
}
else if (大调整) {
  选项 A: 完成当前版本，创建 revision-task
  选项 B: 重置任务，重新开始
}
```

### 待开始任务

```
直接更新:
  - 描述/验收标准
  - spec_version 更新
```

## 当前场景分析

### CR26051701 任务状态

| 任务 | 状态 | 是否满足 DoD | 修订处理 |
|------|------|-------------|---------|
| Phase 0: 环境清理 | ✅ 完成 | ✅ 满足 | 无需修改 |
| Phase 1: 创建 Skill | ✅ 完成 | ❌ 有缺陷 | **创建 revision-task** |
| Phase 1: 更新文档 | ✅ 完成 | ❌ 有缺陷 | **创建 revision-task** |
| Phase 1: 创建模板 | ✅ 完成 | ✅ 满足 | 无需修改 |

### 结论

**应该创建 revision-task，而不是 reopen 已完成的任务。**

原因：
1. 已完成的任务已满足基本 DoD（功能已实现）
2. 问题是设计缺陷，不是实现缺陷
3. 创建 revision-task 更清晰，便于追溯

## Beads 命令示例

### 创建 revision-task

```bash
bd create "修订：修正命令实现方式" \
  --parent {epic-id} \
  --spec-id "CR26051701" \
  --metadata '{"type": "revision", "version": "v1.1"}'
```

### 创建子任务

```bash
bd create "创建命令目录结构" \
  --parent {revision-task-id} \
  --spec-id "CR26051701" \
  -p 1 -t task

bd create "修正 Skill description" \
  --parent {revision-task-id} \
  --spec-id "CR26051701" \
  -p 1 -t task
```
