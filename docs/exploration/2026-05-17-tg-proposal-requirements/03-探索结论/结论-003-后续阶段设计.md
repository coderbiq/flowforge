# 结论-003：后续阶段设计

**状态**: 已确认
**决策时间**: 2026-05-17

---

## 决策

提案/实施/归档阶段参考 OpenSpec 的实现模式，但使用 Skill 而非独立 CLI。

---

## 提案阶段 (`/tg:propose`)

**工件结构**：
```
docs/proposals/CR{编号}-{name}/
├── .proposal.yaml     # 元数据
├── proposal.md        # What & Why
├── design.md          # How
└── notes.md           # 实施笔记（apply 时创建）
```

**核心行为**：
1. 生成提案编号
2. 创建目录结构
3. 填充 proposal.md（渐进式）
4. 询问是否创建任务 Epic
5. 存储长期记忆

---

## 实施阶段 (`/tg:apply`)

**核心行为**：
1. 读取 proposal.md
2. 解析 Capabilities 部分
3. 为每个能力创建任务
4. 创建 notes.md
5. 更新提案状态为 Active

---

## 归档阶段 (`/tg:archive`)

**核心行为**：
1. 检查所有任务是否完成
2. 解析关联模块
3. 更新模块文档（tg-proposal 特有）
4. 移动提案到 completed/
5. 关闭任务 Epic
6. 更新长期记忆

---

## 与 OpenSpec 的差异

| 方面 | OpenSpec | tg-proposal |
|------|----------|-------------|
| 执行方式 | CLI 命令 | Skill |
| 任务管理 | tasks.md | Beads |
| 归档额外操作 | 无 | 更新模块文档 |
| 长期记忆 | 无 | Memory MCP |
