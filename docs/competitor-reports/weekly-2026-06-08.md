### 3.1 BMAD-METHOD v6.8 — 知识持久化的先行者 ⚠️ 重点关注

| 维度 | 评估 |
|------|------|
| **基本信息** | GitHub: `chelleboyer/BMAD-METHODv6`（后迁移至 `bmad-code-org`），v6.8 为当前最新稳定版。21 个专业化 AI Agent 协作框架。 |
| **知识沉淀** | **赛道首创** `.decision-log.md` 决策日志模式——每个决策和覆写都被持久化记录。从 v4/v5 的「会话级记忆」升级为「文件级持久化知识」。PRD→SPEC 内核（Problem/Capabilities/Constraints/Non-goals/Success signal）形成知识提炼链路。但仍不支持 archive 合成——决策日志是追加而非融合。 |
| **知识组织** | 按开发阶段组织（Brief→PRD→SPEC→UX），非系统维度。Web Bundles 支持"Plan in Web, Build in IDE"，消除了 Web 与 IDE 之间的知识断层。 |
| **任务管理** | 21 个 Agent 按角色分工（Analyst/PM/Architect/Developer），TOML 覆写系统支持按项目定制 Agent 行为。但无状态机任务管理，无原子操作。 |
| **工作循环** | 线性门控（Brief→PRD→SPEC→UX→Build），Fast 路径和 Coaching 路径二选一。无中途回退机制。 |
| **反馈机制** | Amelia（开发者 Agent）的 `bmad-investigate` 支持 bug 分类和根因分析（证据分级：Confirmed/Deduced/Hypothesized），但无结构化反馈路由。 |
| **多项目** | 不支持 monorepo 多 subsystem。 |

**与 FlowForge 重叠度：中高（5/6 维度有接触，但均不如 FlowForge 深入）**

BMAD 的 `.decision-log.md` 是本周最重要的行业信号——知识持久化正从 FlowForge 的独门优势变为赛道共识。但其决策日志仍是「追加式记录」而非「合成式融合」，与 FlowForge 的 archive SKILL（create/replace/merge）有本质区别。

| 维度 | 评估 |

## 四、趋势观察

### 4.1 SDD 成熟度三级模型被广泛采纳

社区（特别是 RSMuthu/SDD-comparison 项目）总结的 SDD 成熟度三级模型成为本周讨论热点：

- **Level 1: Spec-First** — Spec 先于编码，随后被丢弃（Spec-Kit、Kiro、BMad）
- **Level 2: Spec-Anchored** — Spec 持久化并演化（OpenSpec、Spec Kitty）
- **Level 3: Spec-as-Source** — 仅编辑 Spec，代码自动生成（Tessl）

FlowForge 不在此分类中，因为它超越了 spec 本身的持久化——FlowForge 关注的是**从 spec 中提取系统性知识并持续沉淀**，可被视为「Level 2+」或独立维度。

### 4.2 知识持久化成为下一个战场

本周多个信号表明知识持久化正成为赛道共识：

- BMAD v6.8 引入 `.decision-log.md`（首个显式知识持久化机制）
- Kilo Code Memory Bank 被更多开发者关注
- OpenSpec v1.4.0 默认启用 sync 技能（规范同步）
- Superpowers 的 worktree 技能重写隐含了"跨会话上下文保持"的需求

**这对 FlowForge 既是验证也是压力**——方向正确，但先发优势窗口在收窄。

### 4.3 多 Agent 编排 vs. 架构扁平化

本周出现了有趣的张力：

- **Superpowers v5.1.0** 将架构完全扁平化（零命名 Agent，全部通用调度），性能提升显著（25min→30sec）
- **BMAD v6.8** 坚持 21 个专业化 Agent 的精细分工
- **GSD-2** 则以编排器 + 子 Agent 的层级模式运行

FlowForge 的 SKILL 路由模式更接近 Superpowers 的扁平化方向（通用 Agent + 专用提示词），但保留了结构化的场景分类。Superpowers 的实践证明这种模式在性能上有显著优势。

### 4.4 社区讨论热点

- **SDD = 新瀑布流？** Hacker News 持续讨论 SDD 是否在重新引入瀑布式官僚流程。社区共识：关键在于「轻量 spec」vs「厚重 spec」的平衡。
- **YC W25 数据**：25% 的项目有 95% AI 生成代码，行业预测 2027 年开发者不再直接查看代码——规范正成为「编程的基本单元」。
- **Dynamic Workflows 冲击**：Anthropic 5月28日发布的 Dynamic Workflows 与 SDD 方法论产生直接编排冲突，Superpowers 社区正在激烈讨论应对方案——这是整个 SDD 赛道需要关注的范式级事件。

### 4.5 国内赛道

国内 SDD 工具生态仍处于早期，以翻译/教程为主（OpenSpec 中文文档、Spec-Kit 实战文章），未见有影响力的国产 SDD 工具出现。这是 FlowForge 的潜在窗口。

### 5.1 短期（本周）

1. **深入分析 BMAD `.decision-log.md` 机制**：这是赛道内最接近 FlowForge 知识沉淀理念的实践。研究其决策日志格式、与 PRD/SPEC 的关联方式、跨会话上下文恢复机制，评估是否值得在 FlowForge 的 decision-log 模板中借鉴。
2. **关注 Superpowers Dynamic Workflows 冲突的解决方案**：这是整个 SDD 赛道面临的架构级问题——编排权归属。FlowForge 的 SKILL 路由 + feedback 循环模式可能对此有天然优势，可在社区讨论中输出观点。
3. **调研 Spec Kitty 的 Worktree 集成模式**：评估是否可在 FlowForge 中引入类似的并行功能隔离机制。

### 5.2 中期（1-2 周）

1. **输出 FlowForge 差异化定位文章**：基于本周收集的 SDD 成熟度三级模型，明确 FlowForge 的「Level 2+」定位——超越 spec 持久化，进入知识合成与系统化沉淀。在 Hacker News / 掘金 / 知乎等平台建立声量。

1. **Tessl 的 spec-as-source 路线**值得持续跟踪——如果 AI 代码生成质量持续提升，spec-as-source 可能成为终局形态。FlowForge 的 archive SKILL 对此有天然适配（spec 变化 → 知识自动更新）。
2. **国内 SDD 工具的空白窗口**可能不会持续太久——GitHub Spec-Kit 的中文社区正在快速壮大，建议在窗口关闭前建立 FlowForge 的中文开发者心智。
