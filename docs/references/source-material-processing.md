# 长文资料处理：方法论综述

> 本文档调研 Zettelkasten 社区和知识管理领域对"长文拆解为原子知识"的方法论，为 FlowForge 外部资料导入 SKILL 提供设计基础。

## 1. 核心问题

外部资料（SKILL 文件、工程指南、遗留文档、参考文章）是长文形式。要进入 FlowForge 的 library 知识库，必须拆解为原子卡片。这需要回答：

1. 如何从长文中识别出独立的"知识单元"？
2. 如何为一个知识单元确定合适的卡片类型？
3. 如何组织多张卡片形成可导航的知识网络？
4. 如何避免信息丢失或过度拆分？

## 2. Zettelkasten：文献笔记 → 永久笔记

### 2.1 Ahrens 的核心流程

Sönke Ahrens 在 *How to Take Smart Notes* 中定义了从阅读到知识的完整流程：

```
阅读 →  fleeting notes（闪念笔记）
     →  literature notes（文献笔记：用自己的话重述，附来源引用）
     →  permanent notes（永久笔记：脱离来源，融入知识网络）
```

**关键步骤**：

1. **阅读时**：用笔在手，标记段落，写下简短批注（只够重建心理上下文即可）
2. **处理后**：回头审视批注，用自己的话重新表述核心观点
3. **拆解**：不照搬原文结构，而是按概念重新组织
4. **连接**：每张永久笔记创建时必须至少链接到一张已有笔记

### 2.2 对 FlowForge 的启示

- `literature notes` 对应 `source material` 的登记（记录来源，标记出处）
- `permanent notes` 对应 library 中的原子知识卡（convention / module / decision / finding）
- 关键门槛：**必须用自己的话重述**，不能复制粘贴原文

## 3. Progressive Summarization：渐进式压缩

### 3.1 Tiago Forte 的五层模型

Tiago Forte 在 *Progressive Summarization* 中提出"机会主义压缩"——在多次回顾中逐步压缩信息：

| Layer | 操作 | 说明 |
|-------|------|------|
| 0 | 原始全文 | 来源文档 |
| 1 | 摘录 | 捕获感兴趣的部分 |
| 2 | 粗体 | 标记摘录中最好的部分 |
| 3 | 高亮 | 标记粗体中"最好的最好的" |
| 4 | 执行摘要 | 用自己的话写摘要 |
| 5 | 再创作 | 转化为博客、设计、演讲 |

### 3.2 核心洞察

- **压缩与上下文的权衡**：压缩太多失去意义，压缩太少失去可发现性
- **机会主义压缩**：不要求一次性完成，而是在每次回顾时做一点
- **Note-first 设计**：把单条笔记的设计作为首要因素，而非依赖标签或文件夹结构
- **Future You 是苛刻的客户**：笔记必须让未来的自己能快速判断是否值得读

### 3.3 对 FlowForge 的启示

- 外部资料导入不需要一次性完美拆解。可以先用 `status: draft` 创建卡片，后续迭代完善
- 每张卡片需要在"压缩"（摘要）和"上下文"（正文）之间取得平衡
- SKILL 的导入计划就是 Layer 4（执行摘要）——在真正写入卡片前先产出可审查的摘要

## 4. Evergreen Notes：收集 → 处理 → 连接

### 4.1 Andy Matuschak 的循环模型

Andy Matuschak 的 Evergreen Notes 方法论强调"阅读时轻量收集，事后深度处理"：

```
收集阶段（阅读时）
  -> 标记段落，写简短批注（只够重建上下文）
  -> 不切换到"写笔记"模式，保持阅读流畅

处理阶段（事后）
  -> 聚类：将批注按概念分组（不按原文结构）
  -> 写宽泛笔记：捕捉一个聚类的"大想法"
  -> 写细粒度笔记：捕捉聚类中的原子想法
  -> 连接：搜索已有笔记，建立链接、合并、修订
  -> 修订：回到宽泛笔记，基于细粒度笔记改善摘要
  -> 循环
```

### 4.2 核心原则

- **概念导向**：按概念组织笔记，不按作者/书名/项目
- **正交于原文结构**：聚类不遵循原文章节顺序，而是围绕任何你认为合适的目的
- **原子性**：每张笔记只讲一件事，但尽可能完整地讲透
- **密集链接**：笔记之间通过链接形成网络，产生涌现洞察

### 4.3 对 FlowForge 的启示

- "聚类"步骤对应 STR 索引卡的设计：一个 STR 卡组织一个概念簇下的 7-15 张原子卡
- "先宽泛后细粒度"对应先创建 STR 索引卡（主题级），再创建具体知识卡（原子级）
- "修订"步骤说明导入不是一次性操作，SKILL 应该支持迭代完善

## 5. Christian Tietze：三阶段处理法

### 5.1 流程

```
Phase 1: Pull the notes
  -> 从书中拉出所有批注，不关心顺序

Phase 2: Form clusters and create overviews
  -> 将批注按主题聚类（正交于书的结构）
  -> 相同的概念可能出现在书的不同章节，聚类把它们聚合在一起

Phase 3: Write notes from clusters
  -> 每个聚类先写最概括的笔记
  -> 再拆出需要前置知识的细节笔记
  -> 细节笔记链接回概念笔记
  -> 根据细节笔记改善概念笔记的摘要
```

### 5.2 核心原则

- **正交于内容**：聚类不遵循原文章节顺序，而是围绕任何你认为合适的目的
- **原子性促进复用**：分离关注点，使得论证可以脱离假设单独复用
- **自顶向下 + 自底向上结合**：先建概念笔记（自顶向下），再拆细节（自底向上），最后回到概念笔记修订

## 6. 方法论综合：输入 → 处理 → 输出

综合以上方法论，外部资料导入的通用模式可以抽象为：

```
输入：长文资料
  |
  v
阶段 1：登记来源
  - 记录来源文档路径、标题、类型
  - 创建 source material 登记卡（可选，用于追溯）
  |
  v
阶段 2：提取知识单元
  - 识别长文中可独立存在的知识断言
  - 标注每个知识单元的类型（convention / decision / finding / principle / pattern / fact / example）
  - 用自己的话重述（不复制粘贴）
  |
  v
阶段 3：聚类
  - 将知识单元按概念聚类（不按原文结构）
  - 每个聚类对应一个主题
  - 识别聚类内知识的依赖关系
  |
  v
阶段 4：生成导入计划
  - 每个聚类 → 一个 STR 索引卡
  - 每个知识单元 → 一张原子知识卡
  - 标注重复/合并候选
  - 标注过大或模糊的卡片警告
  - 用户审查
  |
  v
阶段 5：写入 library
  - 使用 CLI 创建卡片（status: draft）
  - 建立卡片间链接
  - 建立 STR 索引
  - 重建 sqlite 索引
```

## 7. 对 FlowForge SKILL 设计的直接指导

### 7.1 知识单元的类型判定

综合 Zettelkasten 社区和 FlowForge 的知识类型讨论，一个知识单元可以判定为以下类型之一：

| 类型 | 判定标准 | 示例 |
|------|----------|------|
| `convention` | 约束未来工作的规则 | "分页查询必须传 pageSize" |
| `decision` | 已接受的技术/产品决策 | "使用 Redis 做缓存层" |
| `finding` | 可复用的事实或注意事项 | "npm link 在部署中不可靠" |
| `principle` | 指导性原则 | "SKILL 是薄适配器" |
| `pattern` | 可复用的设计模式 | "Command 模式处理多操作场景" |
| `fact` | 客观事实 | "API 网关限制 10MB 请求体" |
| `example` | 示例/案例 | "XX 项目的分页实现方式" |
| `module` | 模块/系统知识 | "auth 模块负责认证和授权" |

### 7.2 聚类策略

- **不按原文章节**：原文的章节结构服务于叙事，不是知识组织的最佳方式
- **按概念**：相同概念可能出现在文档的不同位置，聚类把它们聚合
- **按项目维度**：对于工程文档，按 layer / domain / module 等维度聚类
- **聚类粒度**：一个聚类对应一个 STR 索引卡，包含 3-15 张原子知识卡

### 7.3 原子性判定

一张知识卡是否足够原子？三重判定：

1. **自足性**：脱离上下文能否独立理解？
2. **单一性**：是否只做一个断言，阐述一个概念？
3. **可寻址性**：能否被其他卡片独立引用？

如果"是"×3，则足够原子。如果某条知识需要大量前置知识才能理解，说明它依赖其他卡片，应拆分为：
- 概念卡（解释前置概念）
- 结论卡（引用概念卡，阐述结论）

### 7.4 压缩与上下文平衡

每张知识卡的结构：
- **标题**：一句话概括（高压缩）
- **摘要**（frontmatter summary）：2-3 句话（中压缩）
- **正文**：完整上下文、来源证据、示例（低压缩，保留理解所需的全部上下文）
