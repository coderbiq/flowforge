FlowForge：让 AI 编码不止于「写完即忘」

一个面向 AI 辅助软件开发的方法论工具包——把探索、设计、实施和知识沉淀变成一个可持续生长的闭环

v0.11.2 开源 MIT License

一、我们到底在解决什么问题？

AI 辅助编程已经走过「能不能写代码」的阶段，进入了「写完之后呢？」的阶段。三个尖锐的问题正在浮现：

🔁 知识在新旧对话间断裂

上一轮对话里 Agent 花了 40 分钟搞懂了你们项目的权限模型，下一轮对话，一切从头开始。探索成本反复支付，没有人记得「上次已经搞明白了」。

📋 设计决策散落在对话历史里

为什么选了 Redis 而不是 RabbitMQ 做消息队列？三个月后，没人想得起来。当初的 trade-off 分析、备选方案评估，全部消失在 LLM 的上下文窗口里。

🧩 任务管理在 AI 流程外孤岛运行

Jira 上有一个故事卡，Agent 在执行代码，两套系统互不相通。Agent 不知道 Jira 上的依赖关系，Jira 也不知道 Agent 做到了哪一步。切换成本高得离谱。

💡 核心矛盾：AI 的上下文是瞬时的，但软件工程的知识需要持久。当前的工具生态在「让 Agent 把事做完」上投入很多，在「让 Agent 做完的事被记住」上投入很少。

二、高星竞品在知识管理上的「足够好」陷阱

社区里最火的项目——Superpowers、OpenSpec、Spec-Kit——都在做 AI 辅助开发流程，但它们在知识管理上各自有盲区。下面从四个维度做一次「压力测试」。

2.1 知识持续沉淀：写完的决策，下次还能找到吗？

2.2 知识组织方式：碎片化记录 vs 结构化体系

大多数竞品存在一个结构性问题：知识的组织形式 = 提案的组织形式。这意味着知识是按「变更」维度索引的，不是按「系统」维度索引的。

维度竞品典型做法FlowForge 的做法
索引方式 按变更/提案编号（changes/CR-001/） 按 domain 分类： system 级架构 → library/architecture/；module 级设计 → library/modules/<name>/
文档类型 通用的 spec.md / design.md 12 种 doc_type： proposal, design, module, architecture, convention, adr, decision, finding, notes, journal 等——每种有独立的 writing guide
跨提案关联 无或弱关联 domain frontmatter — 每篇文档标注 {scope, module, type}，自动推导归档路径
知识演进 追加新文件，旧文件不变 replace/merge 策略 — 归档时主动修正 library 中的过时描述（旧命名、旧架构）

2.3 文档管理任务 vs Beads 任务管理：静态表格 vs 状态机

Superpowers 和 OpenSpec 都用文档（plans.md / tasks.md）记录任务。这看起来简单，但有一个被忽视的代价：

❌ 纯文档管理任务的不足

  • 无并发保护：两个 Agent 同时读到同一个 tasks.md，都以为自己是第一个认领的
  • 无状态机：「已完成→重开」需要手动编辑 Markdown，容易出错
  • 无法查询依赖：「有哪些任务依赖已满足、可以开始了？」需要人眼扫描
  • 与 AI 上下文脱节：Agent 不知道这个任务的上一个状态是什么（除非把整个文件塞进上下文窗口）
  • 无法原子操作：「认领并切换状态」不是一个原子动作

✅ Beads 后端的状态机管理

  • 原子操作： flowforge task claim — 认领成功或返回冲突，不会有竞态
  • 完整状态机： pending → in_progress → done / blocked，带来源追溯（sourceTasks）和依赖
  • 就绪查询： flowforge task ready 直接返回依赖已满足的 pending 任务列表
  • 4 层层级： Main Epic → Type Sub-Epic → Task → Child Task，父子任务缩进展示
  • 快照分离： 数据存在 Beads 后端，tasks.snapshot.md 只是自动生成的只读展示（供 git diff 审查）

三、FlowForge 的方法论：六项设计原则

FlowForge 不是一个「怎么做」的操作手册，而是一套组织原则——它告诉 AI Agent 如何把人类的需求理解、方案设计、代码实施、经验沉淀组织成一个可持续运转的循环，而不是一次性的对话。

1. 需求树驱动，而非阶段驱动

不是「先分析完所有东西，再开始设计」。需求树是一棵在探索中逐渐长全的活地图——初始只有草案（不完整的节点标 [?]），每完成一个分析任务，确认或移除一个不确定性，树就完整一分。设计和分析交织进行。

2. 探索即沉淀

探索代码库时发现的系统架构事实直接写入 library/，不等 proposal 归档。library 是系统的「当前真相」——命名约定、模块分层、权限模式，发现一个就记录一个。下一轮 design SKILL 自动加载 library 作为上下文。

3. 反馈是第一等公民

实施中遇到的不只是 bug。FlowForge 把反馈分为五种类型：bug（修复任务）、finding（直接写入 library 对应路径）、knowledge（标记待归档提取）、missing-requirement（回退到 design 补充需求）、design-flaw（回退到 design 修改方案）。每种有不同的路由。

4. 归档是合成，不是收档

archive SKILL 的核心动作是 create / replace / merge 三种写入策略——对比 library 现状，用提案中的最新设计修正过时描述。旧的命名、旧的架构，在归档时被主动更新。

5. 机制与策略分离

核心工作流规则在 FlowForge 本体，项目级偏好和约束通过 config.yaml 注入。每个 project 可以定义自己的命名规则、探索策略、设计指导、归档行为，不影响核心 SKILL 的运转。

6. 自洽命中路由

六个 SKILL 各自通过精确的 description 规定触发信号词和排除条件。不依赖中心路由器，Agent 根据用户话语中的信号词自动匹配最合适的 SKILL。每个 SKILL 知道「什么场景该激活，什么场景不该激活」。

四、面向真实项目的扩展能力

4.1 项目级规则扩展：一个仓库，多个 project，各自独立配置

大多数竞品假设「一个仓库 = 一个项目」。但真实世界的 monorepo 场景下，一个仓库可能包含多个子系统，每个子系统有自己的知识库路径、源码目录和规则偏好。

FlowForge 通过两级配置解决：

# .flowforge/config.yaml — 仓库级配置，定义有哪些 project
projects:
  - id: backend
    name: "后端服务"
    config: projects/backend.yaml

  - id: frontend
    name: "前端应用"
    config: projects/frontend.yaml
# .flowforge/projects/backend.yaml — project 级配置
wikiRoot: wiki-backend          # 独立的知识库目录
srcDirs: [src/server, src/shared]  # 源码目录映射
keywords: [api, 权限, 数据库, 消息队列]
description: "核心后端服务，负责业务逻辑和数据持久化"

rules:
  design:
    naming:
      proposal_id: "BE{YYMMDD}{NN}"    # 后端专用 proposal ID 格式
    strategy: |
      设计方案时优先考虑数据库兼容性和 API 向后兼容；
      跨模块变更先画出影响范围再进入细节。

  implement:
    strategy: |
      每个 PR 必须通过集成测试；
      数据库迁移使用 Liquibase，禁止手动改 DDL。

  library:
    requireReview: true          # 写入 library 前需要 review
    autoUpdateHistory: true      # 自动追加模块变更历史

📂 多 WIKI 目录映射

每个 project 的 wikiRoot 独立指定。proposal 创建在 <wikiRoot>/workspace/proposals/，知识沉淀到 <wikiRoot>/library/。同一个仓库里，前后端的知识库完全隔离。

📁 多代码目录映射

srcDirs 定义该 project 对应的源码目录。design SKILL 通过引用文件落在哪个 srcDir 下自动匹配 project 归属——命中 srcDir 的双倍权重，命中 keyword 的常规权重。

🧭 项目级策略注入

每个 SKILL 的 strategy 字段接受自然语言项目级指导。exploration 怎么探索代码库、design 怎么做决策、implement 的代码规范、archive 的知识优先级——全部可定制。

📝 校验闸门

每个 doc_type 有独立的 writing guide 和 JSON Schema frontmatter 校验。flowforge validate-doc 确保 library 文档完整性，flowforge validate-proposal 校验 proposal 目录结构。

4.2 部署模型：非 SaaS、非 IDE 插件

FlowForge 是可部署的工具包——./scripts/install.sh <目标项目路径> 一键安装到任何仓库。部署后，目标项目获得：

目标项目/
├── .agents/skills/       ← 6 个 SKILL 文件
├── .flowforge/           ← 配置、schema、guides、hooks
├── ff-wiki/              ← 知识库骨架
└── AGENTS.md             ← 自动追加 FlowForge 路由入口

这意味着：FlowForge 不依赖外部 IDE（不需要安装 VS Code 插件），不依赖 SaaS 平台（不需要创建账号），不依赖特定 Agent 实现。只要目标项目的 Agent 能读取 SKILL 文件，FlowForge 就能工作。

五、未来路线图：让知识不止于「写下来」，还能「被找到」和「被维护」

当前版本（v0.11.2）解决了知识「如何被组织」和「如何被写入」的问题。接下来的挑战是检索和质量。

5.1 library 与项目代码的向量索引

🔮 目标：Agent 在探索阶段不只是「读取 library 文件」，而是能语义搜索整个知识库和代码库——「哪些模块跟这个权限模型有关？」「历史上有哪些类似的方案被讨论过？」

  • 对 library/ 下的所有文档建立向量索引，按 domain (scope + module + type) 组织
  • 对项目源码建立向量索引，按 module/project 组织
  • design SKILL 探索阶段：语义搜索 library → 找到相关架构决策/模块设计/编码约定 → 作为探索上下文注入
  • feedback SKILL 识别发现时：语义搜索 library → 判断当前发现是否与已有知识冲突 → 自动标注为「可能与已有知识不一致」
  • archive SKILL 合成时：语义搜索 library → 找到所有受影响的文档 → 生成更精确的 create/replace/merge 计划

5.2 library 内容的持续质量检查与修正

🔮 目标：自动发现 library 中的过时文档、不一致描述、缺失覆盖，并生成修正建议。

  • 过期检测：当代码结构发生变更（模块重命名、目录重组），自动标记 library 中引用了旧路径/旧命名的文档
  • 一致性检查：对比 library 中同一模块的多篇文档，检测相互矛盾的描述（如 architecture 说用 Redis，但 decisions 里选了 Memcached）
  • 覆盖度评估：扫描所有 srcDir，列出 library 中没有对应 module 文档的源码模块——发现知识盲区
  • 修正建议：结合 code → library 语义匹配，生成「这份 architecture 文档中提到 UserService，但代码中已重命名为 AccountService」级别的修正建议
  • 定期运行：作为 CI 检查或 cron 任务运行，输出质量报告（过期数 / 冲突数 / 盲区数），并为每个问题生成修正任务

📐 质量检查架构设想

flowforge library-check → 扫描 library + srcDirs → 生成三份报告：
① 过期报告（文档引用不存在的路径/模块名）
② 冲突报告（同一模块多文档矛盾描述）
③ 盲区报告（有源码无文档的模块）
→ 每个问题可一键生成修正建议 → 创建 implementation 任务 → 由 Agent 修改

🎯 最终形态

Library 不仅是「写过的东西的仓库」，而是一个自维护的知识系统——随着代码演进，知识也在持续校准。Agent 在每一轮对话中从一开始就拥有对整个系统最准确的理解。

六、总结

FlowForge 解决的不是「如何写出更好的代码」，而是如何让 AI Agent 在持续的软件项目中积累可复用的知识。它不是又一个「写规范的工具」——它是把探索、设计、实施、反馈、归档、索引六个环节串成一个可持续运转的环路。

如果你关心以下问题之一，FlowForge 值得一看：

  • 你的团队在用 AI 辅助编码，但每次新对话都要重新解释项目架构
  • 你有很多设计决策散落在 Slack / 飞书 / 对话历史里，找不到也记不住
  • 你的 monorepo 里有多个子系统，每个都有不同的开发规范和知识结构
  • 你想让知识的积累成为系统性的日常操作，而不是一个「等项目做完再说」的事后反思
